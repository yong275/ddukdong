# 깃허브 협업 가이드 (Claude Code 기준)

## 전체 작업 순서

```
① 처음 시작 (최초 1회)
   레포 클론 · 패키지 설치
         ↓
② 작업 새로 시작 ⭐
   dev 최신 받기 · 새 브랜치 생성
         ↓
③ 코드 작업
         ↓
④ 커밋 ⭐
   변경 파일 스테이징 · 커밋 메시지 작성
         ↓
⑤ 푸쉬 ⭐
   내 컴퓨터 → 깃허브 서버에 올리기
         ↓
⑥ PR 하기 ⭐
   dev 최신 머지 → 충돌 해결 → PR 올리기
         ↓
⑦ PR 이후
   본인이 머지 → 브랜치 삭제 → ②로 돌아가 반복
         ↓
⑧ dev 테스트 완료 후
   dev → main PR · 배포
```

---

## 브랜치 전략

```
main        → 배포용. 직접 푸쉬 금지.
dev         → 개발 통합. PR로만 병합.
feature/*   → 기능 개발.
fix/*       → 버그 수정.
```

### 브랜치 흐름
```
feature/solar-api
      ↓ PR
     dev
      ↓ PR (테스트 완료 후)
     main
```

---

## 커밋 메시지 규칙

```
feat:     새 기능 추가
fix:      버그 수정
refactor: 코드 리팩토링 (기능 변화 없음)
style:    스타일·포맷 변경
docs:     문서 수정
chore:    설정·패키지 변경
```

### 예시
```
feat: Solar API 스토리 생성 연동
fix: 이미지 URL null 오류 수정
docs: 깃허브 협업 가이드 추가
chore: dotenv 패키지 추가
```

```

"feat: Solar API 스토리 생성 연동" 으로 커밋해줘
```

## ⭐ PR 하기 (dev에 병합)

**1단계 · dev 최신 내용 먼저 반영**
```
dev 최신 내용을 현재 브랜치에 머지해줘
충돌 있으면 알려줘
```

**2단계 · 푸쉬**
```
현재 브랜치 origin에 푸쉬해줘
```

**3단계 · PR 올리기**

```
현재 브랜치를 dev로 PR 올려줘
제목은 "[커밋메시지 규칙에 맞게]"
본문은 아래 내용으로 작성해줘

작업 내용: [어떤 기능을 만들었는지]
변경 파일: [주요 변경 파일]
```

예시
```
현재 브랜치를 dev로 PR 올려줘
제목은 "feat: Solar API 스토리 생성 연동"
본문은 아래 내용으로 작성해줘

작업 내용: Solar API 연동해서 나이대별 스토리 텍스트 생성 구현
변경 파일: server/services/solarService.js
```

---

## PR 이후 할 일

**1단계 · 본인이 바로 머지**
```
dev PR 머지해줘
```

**2단계 · 브랜치 삭제**
```
머지된 feature/[브랜치이름] 브랜치 삭제해줘
로컬이랑 원격 둘 다 삭제해줘
```

**3단계 · 다음 작업 시작**

⭐ 작업 새로 시작할 때 섹션으로 돌아가서 반복
